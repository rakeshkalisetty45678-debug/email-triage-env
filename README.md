# Executive Assistant Negotiation Env

`Executive Assistant Negotiation Env` is an OpenEnv-compliant environment for training LLMs on realistic assistant work: negotiating with multiple stakeholders, protecting scarce calendar slots, delegating to the right teammate, and preserving personal commitments under pressure. The repo now also ships with a Streamlit UI for interactive playthroughs and demos.

This submission intentionally targets three hackathon themes at once:

- Theme 1: Multi-Agent Interactions
- Theme 2: Super Long-Horizon Planning and Instruction Following
- Theme 3.2: Personalized Task Handling

## Why this is interesting

Most inbox environments collapse the task to classification. Real assistant work is harder:

- stakeholders have different incentives
- some requests should be handled async, others need live intervention
- delegates have limited capacity and different strengths
- one early action can create downstream failures
- relationship damage matters alongside operational success

This environment trains precisely those tradeoffs.

## Environment design

Each episode places the agent in a partially observable executive workflow. The agent sees:

- the current email thread
- visible constraints and social risk
- limited calendar slots
- teammate capacity and specialties
- stakeholder preference hints
- the running conflict log and prior outcomes

At each step, the agent must emit a structured action:

- `reply`
- `delegate`
- `schedule`
- `decline`
- `clarify`
- `archive`

It must also assign a priority, optionally choose a delegate or slot, and draft a short message.

## Scenarios

### `board_crunch`

Protect a family dinner commitment while handling an investor request, a security escalation, and board-prep work.

### `launch_week`

Coordinate product, legal, sales, and co-founder constraints during launch week without wasting scarce executive time.

## Reward model

The environment uses composable OpenEnv rubrics:

- `DecisionRubric`: was the chosen action type correct or at least acceptable?
- `PriorityRubric`: did the agent calibrate urgency correctly?
- `CoordinationRubric`: did it choose the right delegate or calendar slot?
- `CommunicationRubric`: did the rationale and message mention the right constraints?
- `LongHorizonOutcomeRubric`: did the full trajectory avoid conflicts and resolve the important work?

This produces dense per-step rewards while still preserving delayed reward from end-of-episode outcomes.

## Training pipeline

The repo now includes two TRL paths:

- Minimal RL script for the OpenEnv requirement: [minimal_trl_train.py](scripts/minimal_trl_train.py)
- Faster submission pipeline for before/after learning evidence: [train_sft_submission.py](scripts/train_sft_submission.py)

The recommended submission path is:

```bash
pip install -r requirements.txt
pip install -r requirements-train.txt
python scripts/train_sft_submission.py
python scripts/plot_training_state.py
python scripts/evaluate_submission.py
```

For a much faster local CPU smoke run:

```bash
set FAST_LOCAL_TRAIN=1
python scripts/train_sft_submission.py
python scripts/plot_training_state.py
python scripts/evaluate_submission.py
```

For a stronger final GPU run:

```bash
set FAST_LOCAL_TRAIN=0
set SFT_MODEL_NAME=Qwen/Qwen2.5-0.5B-Instruct
set SFT_REPEATS_PER_SCENARIO=8
set SFT_MAX_STEPS=40
python scripts/train_sft_submission.py
```
